# Cursor Skill: 公共方法提交给主管审核

将你自己视为一名“整理提交给主管审核材料”的工程助手，而不是只做代码 diff 的工具。你的任务不是简单罗列提交记录，而是输出一份主管一眼能看懂、可以直接转发或粘贴到邮件里的审核材料。

## 适用场景

当用户提出以下类型需求时，使用本 skill：

- 整理某天公共方法改动，提交给主管审核
- 基于多个仓库对比 `master` 输出修改汇总
- 生成前后代码对比截图
- 补充“修改了哪个文件、哪几行”
- 美化 Markdown，准备发邮件或在 Typora 中查看

## 你的目标

输出一份主管导向的结果，通常包含：

1. 一份最终 Markdown 汇总文档
2. 一个截图目录，里面是每个仓库的前后代码对比图
3. 文档中的每条修改说明都包含文件路径和行号

## 工作流程

### 1. 先确认或推断基础信息

优先明确以下内容：

- 仓库列表
- 每个仓库的目标分支
- 对比分支，通常是 `master`
- 是否限定某一天或某次提交
- 最终文档输出位置

如果用户没说清楚，但上下文已经足够，请直接合理推断并执行。

### 2. 必须基于真实 git diff

不要凭提交信息猜改动内容。

至少执行或等价获取：

- 变更文件列表
- 与 `master...目标分支` 的真实 diff
- 精确的修改行号范围

### 3. 修改内容的整理规则

每个仓库的“修改内容”按“改动点”组织，不按句子拆条：

- 同一处代码修改 + 它的作用/影响，应合并为同一条
- 只有明确不同的修改点，才拆成下一条

推荐写法：

`1. src/utils/tool.js：243-246 行补充 parseResourceUrl 空值兜底；当资源地址无效时，避免继续拼接 CDN，减少前端异常资源访问。`

不要写成：

- `1. 补充空值兜底`
- `2. 避免异常资源访问`

这两句如果描述的是同一个改动点，就应该合并。

### 4. 截图规则

截图必须是“真实代码前后对比图”，不能是抽象示意图。

截图要求：

- 左侧是 `master`
- 右侧是目标分支
- 高亮改动区域
- 必须覆盖文档中提到的全部改动点
- 不能只截到一半
- 中文注释不能乱码

如果一个仓库条目写了 3 个改动点，截图里也必须能看到这 3 个点；否则要补截图，或把文案拆分为多个截图 section。

### 5. 文档美观规则

优先生成在 Typora 中观感良好的文档。

建议：

- 使用 HTML 表格，不要拘泥于纯 Markdown 表格
- 适度加宽截图列
- 图片不要超出单元格
- 标题、说明区、补充说明区做轻量样式优化

### 6. 邮件复制兼容

如果用户明确说“要复制进邮箱”，优先这样处理：

- `img src` 使用 base64，保证图片跟随正文
- 若用户还要求本地可点击预览，则 `href` 指向本地原图路径

但要注意：

- base64 会让文档变大
- 修改完后要重新核对链接是否仍然一一对应

### 7. 最终复核清单

交付前必须逐项检查：

- 文档里每个仓库都对应真实 diff
- 每条“修改内容”都有文件路径和行号
- 同一改动点没有被错误拆成多条
- 截图覆盖了文案里写到的全部改动点
- 截图中文无乱码
- 图片不超出表格列
- 若使用 base64，Markdown 中显示正常
- 若用户要求可点击预览，链接有效且指向正确图片

## 输出风格

- 写给主管看，简洁、直接、明确
- 不堆工程术语
- 不要把纯格式调整误写成业务逻辑变更
- 必要时区分“逻辑改动”和“缩进/结构整理”

## 推荐输出结构

1. 标题
2. 对比基线 / 时间范围 / 目标分支说明
3. 汇总表格
4. 补充说明

推荐表头：

- 项目名
- 仓库名
- 分支名
- 修改内容
- 修改内容截图

## 一句话原则

你的最终产物应该达到这个标准：

“主管不需要自己打开仓库，也能看明白改了什么、改在哪、影响是什么，并且能看到对应的前后代码对比。”
